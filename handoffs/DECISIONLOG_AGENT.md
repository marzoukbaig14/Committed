# Handoff: Decision-Log Agent

You run the decision-logging system for Committed, and you help the human build it. Read `CLAUDE.md` first; it is your behavioral contract. This document contains both the design of the system and the protocol you follow once it exists.

This system is stood up **first**, before the environment, so that every later decision (including the environment setup) is captured by it. That is deliberate: the project records its own evolution from the very beginning.

## What this is, in one line

An append-only set of Architecture Decision Records (ADRs), plus a small script that turns them into a readable log and a relationship diagram. It is decision control: git versions your code, this versions your design and dev choices, the ones git does not capture well, like why you picked this model or split the dependencies this way.

Teach the human: ADRs are a standard, recognized engineering practice, so this is adopting a convention, not inventing one. Each record captures one decision with its context, the decision itself, and its consequences. You never delete a record. When a decision changes, you write a new record that supersedes the old one, and the old one stays in place with its status updated. That preserves the trail, which is the whole point.

## File layout

```
docs/
├── decisions/                 # one markdown file per decision (the source of truth)
│   ├── 0001-adopt-adr-logging.md
│   ├── 0002-...
│   └── ...
├── DECISION_LOG.md            # generated table, never hand-edited
└── decision-tree.md           # generated Mermaid diagram, never hand-edited
scripts/
└── build_decision_log.py      # reads the records, writes the two generated files
```

## The record format

Each record is a markdown file named `NNNN-short-slug.md` (zero-padded four-digit id) with YAML frontmatter. Use this template:

```markdown
---
id: 0000
title: <short imperative title>
date: YYYY-MM-DD
status: proposed        # proposed | accepted | superseded | deprecated
supersedes: []          # ids this replaces, e.g. [0003]
superseded_by: []       # filled in when a later record replaces this one
relates_to: []          # non-superseding links, e.g. [0002]
tags: []                # e.g. [model, data, serving, infra, eval, scope]
---

## Context
The situation, the forces in play, the problem being solved.

## Decision
What was decided. State it plainly.

## Consequences
What this makes easier, what it makes harder, what the project is now committed
to, and anything to revisit later.
```

## The generator script

Create `scripts/build_decision_log.py` with the contents below. It needs `pyyaml`, which is in the CPU dependency group. Run it with `uv run python scripts/build_decision_log.py`. You may either hand the human this implementation directly or rebuild it with them step by step as a first guided exercise; ask which they prefer.

```python
#!/usr/bin/env python3
"""Regenerate DECISION_LOG.md and decision-tree.md from the ADR records in
docs/decisions/. Run: uv run python scripts/build_decision_log.py

This is a build step. Never hand-edit the two generated files; edit the records
and re-run this script.
"""
from __future__ import annotations

import sys
from pathlib import Path

import yaml  # from the pyyaml package

ROOT = Path(__file__).resolve().parents[1]
DECISIONS_DIR = ROOT / "docs" / "decisions"
LOG_OUT = ROOT / "docs" / "DECISION_LOG.md"
TREE_OUT = ROOT / "docs" / "decision-tree.md"

LIST_FIELDS = ("supersedes", "superseded_by", "relates_to", "tags")


def parse_record(path: Path) -> dict:
    text = path.read_text(encoding="utf-8")
    if not text.startswith("---"):
        raise ValueError(f"{path.name}: missing YAML frontmatter")
    _, frontmatter, body = text.split("---", 2)
    meta = yaml.safe_load(frontmatter) or {}
    meta["_body"] = body.strip()
    meta["_file"] = path.name
    for key in LIST_FIELDS:
        val = meta.get(key)
        meta[key] = [] if val is None else (val if isinstance(val, list) else [val])
    return meta


def load_records() -> list[dict]:
    records = [parse_record(p) for p in sorted(DECISIONS_DIR.glob("[0-9][0-9][0-9][0-9]-*.md"))]
    records.sort(key=lambda r: str(r.get("id", "")))
    return records


def fmt_id(value) -> str:
    return str(value).zfill(4)


def build_log(records: list[dict]) -> str:
    lines = [
        "# Decision Log",
        "",
        "_Generated by `scripts/build_decision_log.py`. Do not edit by hand._",
        "",
        "| ID | Date | Status | Title |",
        "|----|------|--------|-------|",
    ]
    for r in records:
        rid = fmt_id(r.get("id"))
        lines.append(
            f"| [{rid}](decisions/{r['_file']}) | {r.get('date', '')} "
            f"| {r.get('status', '')} | {r.get('title', '')} |"
        )
    return "\n".join(lines) + "\n"


def build_tree(records: list[dict]) -> str:
    lines = [
        "# Decision Tree",
        "",
        "_Generated by `scripts/build_decision_log.py`. Do not edit by hand._",
        "",
        "```mermaid",
        "graph TD",
    ]
    for r in records:
        rid = fmt_id(r.get("id"))
        title = str(r.get("title", "")).replace('"', "'")
        lines.append(f'  n{rid}["{rid}: {title}"]:::{r.get("status", "proposed")}')
    for r in records:
        rid = fmt_id(r.get("id"))
        for sup in r.get("supersedes", []):
            lines.append(f"  n{fmt_id(sup)} -. superseded by .-> n{rid}")
        for rel in r.get("relates_to", []):
            lines.append(f"  n{rid} --- n{fmt_id(rel)}")
    lines += [
        "  classDef proposed fill:#fff7e6,stroke:#d4a017;",
        "  classDef accepted fill:#e6f4ea,stroke:#137333;",
        "  classDef superseded fill:#f1f3f4,stroke:#9aa0a6,color:#5f6368;",
        "  classDef deprecated fill:#fce8e6,stroke:#c5221f,color:#5f6368;",
        "```",
        "",
    ]
    return "\n".join(lines)


def main() -> int:
    if not DECISIONS_DIR.exists():
        print(f"No decisions directory at {DECISIONS_DIR}", file=sys.stderr)
        return 1
    records = load_records()
    LOG_OUT.write_text(build_log(records), encoding="utf-8")
    TREE_OUT.write_text(build_tree(records), encoding="utf-8")
    print(f"Wrote {LOG_OUT.relative_to(ROOT)} and {TREE_OUT.relative_to(ROOT)} from {len(records)} records.")
    return 0


if __name__ == "__main__":
    raise SystemExit(main())
```

This is intentionally simple. It is the seed of the "decision control" idea the human parked for later, so it is fine for it to be small and extendable.

## The protocol you follow on every decision

When a design or dev decision comes up (model, data, stack, architecture, scope, infrastructure), do this, in order:

1. **Confirm.** State plainly what is changing and why, and ask the human to confirm before you write anything. If they have not actually decided, do not log it; help them decide first.
2. **Log.** On confirmation, create a new record file with the next id and correct links. If it replaces an earlier decision, set `supersedes: [old_id]` on the new record, and edit the old record so its `status` becomes `superseded` and its `superseded_by` lists the new id. (This status edit is the only kind of edit you make to an existing record. You never delete one.)
3. **Regenerate.** Run `uv run python scripts/build_decision_log.py` and confirm the log and tree rebuilt.
4. **Propagate.** If the decision changes anything in `MASTER.md` or `ROADMAP.md`, update those documents to match. **Never edit `CLAUDE.md`**; it is frozen.
5. **Summarize.** Tell the human what was logged and what files changed.

Routine code commits do not get a record; git handles those. Only design and dev decisions are logged here.

## Seed records to create on day one

Start the log populated with the decisions already made, so the system is immediately useful and demonstrates the format. Two are written out in full below as concrete examples. Create the rest from the table that follows, fleshing out context, decision, and consequences with the human (good practice for them, and a fast way to internalize the format).

### Example, written in full: `0001-adopt-adr-logging.md`
```markdown
---
id: 0001
title: Adopt ADR-based decision logging with a generator
date: 2026-05-29
status: accepted
supersedes: []
superseded_by: []
relates_to: []
tags: [infra, process]
---

## Context
The project will make many design and dev decisions over several weeks, often
across different sessions and agents. Git versions code well but does not capture
why a design choice was made, and those reasons are easy to lose. A prior project
used an append-only decision file and it made the work much easier to track.

## Decision
Adopt Architecture Decision Records: one markdown file per decision in
docs/decisions/, with YAML frontmatter. A script regenerates a human-readable log
and a Mermaid relationship tree from the records. Records are append-only; a
decision is changed by superseding it, not by deleting it.

## Consequences
Every significant decision is now traceable, and the tree shows how decisions
relate and evolve. It costs a small amount of bookkeeping per decision. It also
doubles as a portfolio signal of professional practice, and as a seed for a
possible standalone "decision control" tool later.
```

### Example, written in full: `0003-base-model-qwen3-1p7b.md`
```markdown
---
id: 0003
title: Use Qwen3-1.7B as the primary base model
date: 2026-05-29
status: accepted
supersedes: []
superseded_by: []
relates_to: [0002]
tags: [model, training]
---

## Context
Earlier planning selected TinyLlama-1.1B, chosen for a "make the tiny old model
work" narrative. TinyLlama is a 2023-era model, weak by current standards, and
not code-specialized, which would likely have produced mediocre output. The
binding deployment constraint is CPU-only serving on a free tier, which caps
practical model size at roughly 1 to 2B parameters. Current benchmarks place the
Qwen3 family at the top for small-model fine-tuning, and Qwen has strong code
priors.

## Decision
Use Qwen/Qwen3-1.7B as the primary base, with Qwen/Qwen3-0.6B as an in-family
fallback if CPU serving latency is unacceptable. Apache 2.0 license. Its native
thinking modes also set up the v2 reasoning ablation.

## Consequences
Better expected output quality than TinyLlama and a current, defensible choice.
The earlier TinyLlama selection is abandoned. Serving code is unaffected by the
1.7B-to-0.6B fallback. The "make a weak model work" narrative is replaced by the
stronger "small model distilled to match much larger ones on a narrow task."
```

## Definition of done for standing up the system
- `docs/decisions/` contains the seed records.
- `scripts/build_decision_log.py` exists and runs.
- `docs/DECISION_LOG.md` and `docs/decision-tree.md` are generated and committed.
- The human understands the confirm-then-log-then-update protocol and can read the tree.
